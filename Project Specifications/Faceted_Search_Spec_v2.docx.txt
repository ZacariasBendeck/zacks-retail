1. Executive Summary
This spec targets the actual Paperclip monorepo codebase. The database already has 20+ reference tables for shoe attributes, and the storefront already has a working faceted filter sidebar with cross-facet exclusion counting. The work required is extending what exists, not building from scratch.


What already works: 6 facets (department, category, brand, size, color, material) with cross-dimensional counting via buildFacetConditions() in publicProductService.ts. The FacetedFilters.tsx component renders these with counts and zero-removal.
What needs to be added: 9 additional facets (shoe type, heel height, heel shape, toe shape, occasion, finish, width, pattern, closure type) plus multi-select within each facet and price range buckets.
What needs to change: Single-select filter behavior must become multi-select (OR within a category, AND across categories). The buildFacetConditions function needs to accept arrays instead of single values.

3.3 Add New Facet Count Queries to getProductFacets()
File: apps/api/src/services/publicProductService.ts
Follow the exact same pattern used for brands and colors. For each new facet, add a query block. Example for shoe type:
// Shoe types — exclude shoeTypeId filter
const stWhere = buildFacetConditions(filters, 'shoeTypeId');
const shoeTypes = db.prepare(`
  SELECT rst.id, rst.name, COUNT(DISTINCT s.style) as count
  FROM skus s
  JOIN ref_shoe_types rst ON rst.id = s.shoe_type_id
  WHERE ${stWhere.conditions.join(' AND ')}
  GROUP BY rst.id, rst.name
  HAVING count > 0
  ORDER BY count DESC
`).all(...stWhere.values);


Repeat this pattern for: ref_heel_heights, ref_heel_shapes, ref_toe_shapes, ref_occasions, ref_finishes, ref_width_types, ref_patterns, ref_closure_types.
3.4 Add Price Range Buckets
The current facet returns only min/max price. Add bucket counts:
const priceWhere = buildFacetConditions(filters);
const priceBuckets = db.prepare(`
  SELECT
    CASE
      WHEN s.price < 25 THEN '$0-25'
      WHEN s.price < 50 THEN '$25-50'
      WHEN s.price < 100 THEN '$50-100'
      WHEN s.price < 200 THEN '$100-200'
      ELSE '$200+'
    END AS label,
    MIN(s.price) as min, MAX(s.price) as max,
    COUNT(*) as count
  FROM skus s
  WHERE ${priceWhere.conditions.join(' AND ')}
  GROUP BY label
  HAVING count > 0
  ORDER BY MIN(s.price)
`).all(...priceWhere.values);

5. Multi-Select Logic (OR Within, AND Across)
This is the key behavioral change. Currently the system allows only one selection per facet. The Zappos model allows multiple:


Within a facet (OR): Selecting both FORMAL and CASUAL in the department filter shows SKUs that are in FORMAL OR CASUAL. The SQL uses IN ('FORMAL', 'CASUAL').
Across facets (AND): Selecting FORMAL in department AND Kisses in brand shows SKUs that are FORMAL AND Kisses. The SQL uses WHERE department IN ('FORMAL') AND brand_id IN (123).


The cross-facet exclusion rule still applies exactly as before:
* When counting department options: Apply all filters EXCEPT department. So the user sees the full list of departments and their counts considering only the other active filters.
* When counting brand options: Apply all filters EXCEPT brand. So the user sees how many styles each brand has given the active department/color/etc. filters.
* Options with 0 matching results are omitted (HAVING count > 0 already does this).
6. Edge Cases
6.1 NULL Reference IDs
Many SKUs imported from RICS will have NULL values for new-ish attributes (e.g., occasion_id, pattern_id). Rules:
* A SKU with occasion_id = NULL appears when no occasion filter is active.
* A SKU with occasion_id = NULL does NOT appear when any occasion filter is active.
* NULL values do NOT appear as a facet option (the JOIN to the ref table naturally excludes them).
6.2 Empty Facet Groups
If all SKUs in the current result set have NULL for a given attribute (e.g., no SKU has a pattern_id set), the facet query returns an empty array. The frontend should hide that entire facet section rather than showing an empty collapse panel.
6.3 Price Bucket Boundaries
$25.00 exactly goes in the $25-50 bucket (WHEN price < 25 fails, falls to WHEN price < 50). $200.00 goes in the $200+ bucket.
6.4 Performance
Adding 9 more facet count queries means ~15 queries per request. With SQLite WAL mode and the existing indexes on all _id columns, this should stay well under 200ms for the current dataset. If it becomes slow, consider a config-driven approach where facet queries are defined in an array and iterated, keeping the code DRY.
