## Intent

`hr` is Odoo's foundational human-resources module: a central `hr.employee` record (~100 fields â€” HR-lite, payroll-adjacent, personal, bank, visa), hierarchical departments, job positions, and a temporal version table (`hr.version`) that captures how an employee's contract and salary evolve over time. It is marked `application=True` â€” an Odoo *app*, not a bridge addon. Every downstream HR addon (attendance, payroll, time-off, recruitment, expense) builds on this schema.

## Dependencies

- `base_setup` â€” onboarding wizard plumbing.
- `digest` â€” periodic admin-summary emails.
- `phone_validation` â€” international phone formatting.
- `resource_mail` â€” links to `resource.resource` + `resource.calendar`, Odoo's generic primitives for anything with working hours (employees, meeting rooms, machines).
- `web` â€” baseline client.

Layering: `hr.employee` is conceptually `resource.resource` (a bookable thing with a calendar) + a `res.users` link + personal-info + contract/version history.

## Function map

| Method | Model | What it does | Why it matters |
|---|---|---|---|
| `create(vals_list)` | `hr.employee` | Batched per-company creation; generates avatar; creates `res.partner` work-contact; creates first `hr.version`; posts onboarding message to mail thread | Employee creation is orchestrated, not a simple insert. |
| `write(vals)` | `hr.employee` | Syncs user-side fields, creates a new `hr.version` if contract/salary fields change, manages the work-contact link, subscribes the user to the department's discuss channel | User â†” employee â†” version sync is bidirectional. |
| `_search`, `search_fetch`, `fetch`, `get_view` | `hr.employee` | For non-HR users, delegates to `hr.employee.public` | Access-tiering is done at the query layer â€” callers don't branch on permission. |
| `create_version(values)` | `hr.employee` | Creates a new version inheriting previous version's fields | Explicit "effective-date-X edit" path, distinct from `write()`. |
| `_get_version(date)` | `hr.employee` | Returns the version effective on `date` | Time-travel query primitive. |
| `_compute_newly_hired()` | `hr.employee` | Flags employees hired within 90 days; searchable | Drives onboarding-targeted UI. |
| `action_archive()` | `hr.employee` | Archives employee; clears `parent_id` / `coach_id` on subordinates; optionally launches departure-reason wizard | Cleanup-on-leave is cascaded â€” prevents orphan manager links. |
| `action_create_users()` + `action_create_users_confirmation()` | `hr.employee` | Bulk-create login users from selected employees; shows confirmation dialog listing the users before acting | Two-step bulk-UX pattern. |
| `check_no_existing_contract(date)` | `hr.employee` | Raises if another contract already covers `date` | Invariant for non-overlapping temporal segments. |
| `_compute_complete_name()` + `parent_path` | `hr.department` | Indexed materialised path for department tree | Pattern for hierarchical trees in Postgres. |
| `write()` manager cascade | `hr.department` | When a department's `manager_id` changes, updates all child employees' `parent_id` | Department-level reassignment. |

## Client surface

Four subfolders under `static/src`: `components`, `fields`, `views`, `core`, plus a top-level `store_service_patch.js` extending the OWL store service. The module contributes custom avatar/card components, custom field widgets (likely the barcode/PIN inputs), and view configurations. Backend client only â€” no POS or storefront surface.

**Tour tests.** `hr_employee_flow.js` scripts the core directory flow: open the HR app â†’ navigate to an employee (Johnny H.) â†’ initiate a chat conversation â†’ open personal preferences. The other three tours cover multi-bank-account setup, public-employee-link redirect, and version-timeline auto-save â€” the version-timeline tour is the UI surface of the temporal-versioning pattern and the most relevant if Zack's ever adopts versioned master-data.

## UI / security / data

18 view XML files â€” mostly per-model CRUD (employee, department, job, category, version, contract-type, work-location, departure-reason) plus `hr_templates.xml` and `res_users.xml` for user-preferences extension. `security/ir.model.access.csv` grants `hr.group_hr_user` read/write on the main directory and `hr.group_hr_manager` on salary / version / bank-account fields â€” tiered access is driven by group membership, not per-field ACLs. Seed data (`hr_data.xml`) includes default job positions, departure reasons, and employee categories.

## Zack's Retail fit

Current state: [`../../modules/employees.md`](../../modules/employees.md) Slice 1 shipped 2026-04-18 (User + Role + Session + argon2id login only). Slice 2+ is still spec. Critical commitment from Slice 1: **Employee and User are unified into one entity** â€” Slice 2 will hang Employee fields off `User`, not split into a separate Employee table with an FK. This rejects Odoo's fundamental split (`res.users` â†” `hr.employee`), so most of `hr`'s architecture does not port 1:1.

| Pattern | Zack's module | Verdict | Port sketch |
|---|---|---|---|
| Access-tiered public employee view (non-HR callers query a read-only view that omits sensitive fields) | `employees` | **Steal at Phase B** | Postgres view `public.employee_public` selecting `id, displayName, salespersonCode, active, homeStoreId` from `User` (no `passwordHash`, no `commissionRate`, no PIN hash). Query layer routes reads by permission: `User` for `employees.read_sensitive`, `employee_public` otherwise. Lands with Slice 2 Employee-field addition. |
| Archive cascade â€” on archive, clear subordinates' manager FK + launch departure wizard | `employees` | **Steal at Phase B** | Add `User.terminatedAt` + `User.managerId` in Slice 2. Service `terminateEmployee(userId)` in `apps/api/src/services/employeeService.ts` sets `terminatedAt`, nulls `managerId` on any subordinate, writes `EmployeeAudit` row. Optional UI prompt for departure reason. |
| Non-overlapping temporal-segment check (`check_no_existing_contract`) | `employees` | **Steal at Phase B** | In the `EmployeePeriod` close flow, enforce DB-level non-overlap. Prisma `@@unique` can't express range overlap; add a Postgres `EXCLUDE` constraint using `daterange` on `(userId, [periodStart, lockedThrough])`. File: `apps/api/prisma/migrations/<date>_employee_period_overlap_exclude/migration.sql`. |
| `newly_hired` computed flag (hired within N days) | `employees` | Watch | Drives onboarding UI (welcome banner, suppress commission-rate defaults until HR sets them). Trivial compute. Add when Slice 2 lands Employee fields. |
| Bulk user creation with two-step confirmation (preview list â†’ confirm â†’ commit) | `employees` | Watch | Applies to the legacy RICS salespeople-import flow deferred to Slice 2. Pattern: `POST /api/v1/users/bulk-preview` returns what would be created; `POST /api/v1/users/bulk-commit` creates after operator ack. |
| Hierarchical department tree with indexed `parent_path` | `store-ops` (org hierarchy if ever) | Watch | Not in current `store-ops` or `employees` scope. Reference for a future "departments" / "buying groups" feature. |
| `hr.version` temporal-snapshot master-data versioning | `employees` | Skip | Zack's already denormalizes rate-at-commit-time into `CommissionLedgerEntry`, solving the "what rate applied to ticket X" need without a version table. Revisit only if audit "what was the employee's full record on date X" becomes a real requirement. |
| Multi-company fields (`company_id` everywhere; `user_uniq` per company) | N/A | Skip | Zack's is single-company. |
| `visa_no` / `work_permit` / `passport_id` / percentage-based bank-account salary distribution | N/A | Skip | Out of scope for Honduran shoe-store retail; European/US HR concerns. |
| `mail.activity.plan` / discuss-channel subscriptions on departments | N/A | Skip | Odoo's mail-activity system is heavy framework baggage; no Zack's analog planned. |

**Recommendation.** The two patterns worth stealing in Slice 2 are (a) the **access-tiered public employee view** â€” it maps cleanly onto the role split between SALESPERSON (needs to see other salespeople on a ticket) and FINANCE (needs to see rates), and (b) the **Postgres `EXCLUDE` constraint for non-overlapping temporal segments** â€” simpler and stronger than anything the current `EmployeePeriod` spec mentions. The single biggest *domain* mismatch is Odoo's Employee-as-separate-table-with-FK-to-User topology: Zack's Slice 1 rejected this, so most of `hr`'s orchestration (creation batching, employee-user sync, version-row generation on write) has no direct analog here. Port the patterns, not the topology.

## Open questions

1. Does the Slice 1 commitment "Employee + User unified into one entity" actually hold for Slice 2, or was it a shipping-week shortcut that gets reversed once HR-lite fields (`commissionRate`, `timeClockPinHash`, `homeStoreId`) start accumulating and the single `User` table gets wide? If the decision holds, the "access-tiered public employee view" verdict still applies â€” just as a view over `User`. If it reverses, re-read this note with Odoo's Employee-plus-User separation back in scope.
2. Is a Postgres `EXCLUDE` constraint with `daterange` on `EmployeePeriod` in the Slice 2 plan, or does the current spec rely on application-level overlap checks? If app-level, confirm that's intentional; the DB-level constraint is strictly stronger.
3. Should `newly_hired` trigger a `platform` notification broadcast (onboarding announcement to the store's Slack-like feed), or is it purely a UI flag? Affects whether `platform` needs a new event type.
